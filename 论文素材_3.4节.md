## 3.4.1 数据集构建与预处理 (Dataset Construction)

### 一、训练数据来源

路由分类器的训练数据来源于 6 个主流问答数据集，每个数据集抽样 500 条开发集样本：

| 数据集 | 类型 | 预期复杂度 |
|--------|------|-----------|
| HotpotQA | Multi-hop | L2 (多跳复杂) |
| Musique | Multi-hop | L2 (多跳复杂) |
| 2WikiMultiHopQA | Multi-hop | L2 (多跳复杂) |
| Natural Questions (NQ) | Single-hop | L1 (单跳事实) |
| SQuAD | Single-hop | L1 (单跳事实) |
| TriviaQA | Single-hop | L1 (单跳事实) |

数据输入格式为 `processed_data/{dataset_name}/dev_500_subsampled.jsonl`。

### 二、数据集标准化结构

为统一处理来源各异的 QA 数据集，本文将所有原始数据预处理为以下标准化 JSONL 格式：

```json
{
  "dataset": "hotpotqa",
  "question_id": "5ac07327554299012d1db5f2",
  "question_text": "Were Scott Derrickson and Ed Wood of the same nationality?",
  "answers_objects": [
    {
      "number": "",
      "date": {"day": "", "month": "", "year": ""},
      "spans": ["yes"]
    }
  ],
  "contexts": [
    {
      "idx": 0,
      "title": "Scott Derrickson",
      "paragraph_text": "Scott Derrickson is an American director...",
      "is_supporting": true
    }
  ]
}
```

各字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `dataset` | string | 数据集来源标识 |
| `question_id` | string | 唯一问题标识符 |
| `question_text` | string | 问题文本 |
| `answers_objects` | list | 标准化答案，支持 `spans`（文本片段）、`number`（数值）、`date`（日期）三种形式 |
| `contexts` | list | 检索上下文文档列表，每条包含标题、段落文本和是否为支持性证据的标记 |

部分数据集包含额外字段：
- **HotpotQA**：`level`（难度：hard/medium/easy）、`type`（类型：bridge/comparison）
- **Musique**：`reasoning_steps`（中间推理步骤链）

这一标准化设计使得下游的标注、训练和评估流程可以完全与数据集来源解耦。

### 三、弱标签提示 (Weak Label Hinting)

在训练数据生成阶段（`classifier/data/generate_training_data.py`），本文提出 **弱标签提示 (Weak Label Hinting)** 策略。具体做法是在调用 LLM（GPT-4o）进行标注时，将数据集本身的类型信息作为上下文提示嵌入到用户消息中：

```
[Context Hint]: This query comes from a {dataset_type} dataset.
Use this as a hint, but verify it based on the query's actual linguistic structure.
```

其中 `dataset_type` 根据数据集名称映射：
- `hotpotqa`, `musique`, `2wikimultihopqa` → `"multi-hop dataset"`
- `nq`, `squad`, `trivia` → `"single-hop dataset"`

**关键设计**：弱标签提示是 **非约束性的 (non-binding)**。标注模型在收到提示后，仍需根据问题的实际语言结构独立判断复杂度，而非盲目采用提示标签。这一设计在保留数据集先验知识的同时，避免了标签噪声的直接传播。

生成的标签包含三个维度：
1. **复杂度标签 (Complexity)**：L0（无需检索）、L1（单跳事实）、L2（多跳/复杂）
2. **索引策略 (Index Strategy)**：None、Lexical (BM25)、Semantic (向量)、Hybrid（混合）
3. **路由动作 (Action)**：Z（无检索）、S-Sparse（仅 BM25）、S-Dense（仅向量）、S-Hybrid（混合）、M（多跳）

标注模型使用 OpenAI Structured Outputs 接口（模型：`gpt-4o`），以确保输出格式的一致性。并发请求数为 10。

标注输出的结构化 Schema（Pydantic）定义如下：

```python
class RagRoutingAnalysis(BaseModel):
    reasoning: str          # 对查询语言结构、实体特异性和逻辑复杂度的逐步分析
    complexity_label: Literal["L0", "L1", "L2"]
    index_strategy: Literal["None", "Lexical", "Semantic", "Hybrid"]
    action: Literal["Z", "S-Sparse", "S-Dense", "S-Hybrid", "M"]
```

标注后每条训练样本的格式为：

```json
{
  "dataset": "hotpotqa",
  "question_id": "5ac07327554299012d1db5f2",
  "question_text": "Were Scott Derrickson and Ed Wood of the same nationality?",
  "reasoning": "This query compares two entities on nationality, requiring multi-hop lookup...",
  "complexity_label": "L2",
  "index_strategy": "Hybrid",
  "action": "M",
  "answers": [{"number": "", "date": {...}, "spans": ["yes"]}]
}
```

### 四、L0 数据合成

由于原始 QA 数据集中缺少"无需检索"类别的样本，本文通过 `classifier/data/generate_L0_data.py` 合成 L0 类别数据，以平衡训练集的类别分布。

合成的 L0 数据涵盖 **6 个多样性类别**：

| 类别 | 示例 |
|------|------|
| 编码与调试 (Coding & Debugging) | Python、JavaScript、SQL 编程问题 |
| 逻辑谜题与数学 (Logic Puzzles & Math) | 基于推理的数学题 |
| 创意写作 (Creative Writing) | 诗歌、邮件、故事改写 |
| 日常对话与角色扮演 (Chitchat & Roleplay) | 问候、建议 |
| 文本处理 (Text Processing) | 翻译、摘要 |
| 常识问答 (Common Sense) | 通用知识 |

**合成参数**：
- 目标数量：**600 条** L0 样本
- 每批生成：10 条/API 调用
- 生成模型：`gpt-4o-2024-08-06`（推荐版本，质量更优）
- 最大并发：5 请求

### 五、数据合并与最终数据集

L1/L2 标注数据与合成 L0 数据通过以下流程合并：
1. 读取已有训练数据 (`combined_training_data.jsonl`)
2. 读取合成 L0 数据 (`l0_synthetic.jsonl`)
3. **合并并随机打乱** (random shuffle)，确保训练时类别均匀分布
4. 输出最终数据集：`final_finetuning_dataset.jsonl`

#### 训练数据数量统计

| 数据来源 | 样本数 | 标签类别 |
|---------|--------|---------|
| HotpotQA | 500 | 主要为 L2 / M |
| Musique | 500 | 主要为 L2 / M |
| 2WikiMultiHopQA | 500 | 主要为 L2 / M |
| Natural Questions (NQ) | 500 | 主要为 L1 / S-* |
| SQuAD | 500 | 主要为 L1 / S-* |
| TriviaQA | 500 | 主要为 L1 / S-* |
| **QA 数据集小计** | **3,000** | L1 + L2 |
| 合成 L0 数据 | **600** | L0 / Z |
| **最终数据集总计** | **3,600** | L0 + L1 + L2 |

#### 训练/测试划分

```python
dataset_split = full_dataset.train_test_split(test_size=0.1, seed=42)
```

| 划分 | 样本数 | 比例 |
|------|--------|------|
| 训练集 | ~3,240 | 90% |
| 测试集 | ~360 | 10% |

#### 数据文件生成流程

```
6 个 QA 数据集 (各 500 条)
        │  generate_training_data.py + GPT-4o 标注
        ▼
  {dataset}_training.jsonl (各 500 条)
        │  合并
        ▼
  combined_training_data.jsonl (3,000 条, L1+L2)
        │
        │  + generate_L0_data.py 合成
        │  + l0_synthetic.jsonl (600 条, L0)
        │  合并 & 随机打乱
        ▼
  final_finetuning_dataset.jsonl (3,600 条)
        │  train_test_split(0.1, seed=42)
        ▼
  训练集 ~3,240 条 / 测试集 ~360 条
```

### 六、数据质量验证

通过 `classifier/data/validate_generated_data.py` 对生成数据进行质量验证：
- 随机抽取样本，使用相同 LLM API 重新分类
- 对比原始标注与重新分类的一致性（Complexity、Index Strategy、Action 三个维度）
- 质量阈值：
  - ≥ 90%：优秀
  - 75%-90%：良好
  - 60%-75%：一般
  - < 60%：需重新生成

## 3.4.3 实施细节 (Implementation Details)

### 一、分类器训练配置

#### 基座模型

| 参数 | 值 |
|------|------|
| 模型 | **Qwen2.5-3B-Instruct** |
| 最大序列长度 | 2048 tokens |
| 精度 | 自动检测 (支持 BF16 的 GPU 使用 BF16，否则 FP16) |
| 量化 | 否（全精度训练） |

#### LoRA 参数

| 参数 | 值 |
|------|------|

| Alpha | **16** |
| Dropout | 0（为启用 Unsloth 加速） |

#### 训练超参数

| 参数 | 值 |
|------|------|
| 每设备 Batch Size | 4 |
| 梯度累积步数 | 8 |
| **有效 Batch Size** | **32** (4 × 8) |
| 学习率 (Learning Rate) | **1e-4** |
| 学习率调度 | Cosine Annealing |
| Warmup 步数 | 20 |
| 训练轮次 (Epochs) | **2** |
| 优化器 | AdamW-8bit |
| Weight Decay | 0.01 |
| 训练/测试划分 | 90% / 10% (seed=42) |
| 最优模型选择指标 | Loss（保留最优 2 个 checkpoint） |
| 日志记录 | TensorBoard，每 10 步 |

#### Unsloth 加速

使用 **Unsloth** 框架实现训练加速（2-5 倍），具体优化包括：
- `FastLanguageModel.from_pretrained()`：加载模型时启用 Unsloth 优化
- `FastLanguageModel.get_peft_model()`：应用 LoRA 时使用 Unsloth 感知的梯度检查点 (`use_gradient_checkpointing="unsloth"`)
- LoRA Dropout 设为 0 以启用 Unsloth 的额外加速优化

#### 数据格式

训练数据转换为 Qwen 聊天模板格式：
- **System Prompt**：标准 RAG 路由器分类提示
- **User Message**：用户问题
- **Assistant Response**：包含 Analysis、Complexity、Index、Action 四个字段的结构化分类结果
- 使用 `tokenizer.apply_chat_template()` 确保格式正确

#### 多跳检索推理 (Multi-hop, Action=M)

| 参数 | 值 |
|------|------|
| 最大跳数 (MAX_HOPS) | 6 |
| 每跳最大重试 (MAX_RETRIES) | 3 |
| 最终生成最大上下文文档数 | 40 |
| 查询去重阈值 (Jaccard) | ≥ 0.9 |

多跳推理流程：
1. **Stage 0 — 逻辑规划**：LLM 生成逻辑分解链 (Entity A → Property → Entity B → ...)
2. **Stage 1 — 迭代推理循环**：最多 6 跳，每跳生成 Thought & Action（Search[query] 或 Answer[answer]），含证据验证
3. **Stage 2 — 最终答案生成**：聚合所有检索文档（≤40），结合逻辑规划和推理链生成最终答案

## 3.4.4 评估指标 (Evaluation Metrics)

### 一、分类性能指标

路由分类器的分类性能在三分类任务上评估（Z / S / M），主要指标：

1. **Accuracy (3-class)**：分类准确率
2. **F1-score**：各类别 F1 及宏平均 F1
3. **Confusion Matrix**：混淆矩阵，分析各类别之间的误分类情况

分类阶段同时记录：
- **推理速度**：每秒处理问题数 (q/s)
- **动作分布 (Action Distribution)**：各路由动作 (Z, S-Sparse, S-Dense, S-Hybrid, M) 的预测数量分布

### 二、QA 性能指标

QA 性能评估基于 `evaluate/metrics/squad_answer_em_f1.py` 实现，采用以下指标：

#### 1. Exact Match (EM)
- **定义**：二值指标。标准化后的预测答案与标准答案完全一致则为 1，否则为 0。
- **标准化流程**：小写化 → 移除冠词 (a/an/the) → 移除标点 → 合并空白字符

#### 2. F1-score (Token-level)
- **定义**：基于 token 级别的精确率 (Precision) 和召回率 (Recall) 计算的调和平均
- **计算方式**：
  - Gold tokens = `set(normalize(gold_answer).split())`
  - Pred tokens = `set(normalize(predicted_answer).split())`
  - Common tokens = Gold ∩ Pred（通过 `collections.Counter` 计算）
  - `F1 = 2 × Precision × Recall / (Precision + Recall)`

#### 3. Accuracy (Bi-directional Match)
- **定义**：双向字符串匹配，满足以下任一条件即为 1：
  - Gold 答案包含于预测答案中（解决模型输出冗长的情况）
  - 预测答案包含于 Gold 答案中（解决 Gold 答案过度详细的情况）
  - 所有 Gold token 均出现在预测答案中

#### 4. Recall (Token Coverage)
- **定义**：标准化 token 级别的覆盖率
- `Recall = |covered_gold_tokens| / |total_gold_tokens|`

#### 聚合方式
- 对于有多个标准答案的问题，取所有标准答案中的 **最大值** (`metric_max_over_ground_truths`)
- 结果按 **路由动作 (Action)** 分组统计，同时计算 **整体 (Overall)** 指标和 **按数据集** 的指标

输出结构示例：
```json
{
  "dataset_name": {
    "overall": {"em": 0.xx, "f1": 0.xx, "acc": 0.xx, "recall": 0.xx, "count": N},
    "by_action": {
      "Z": {"em": ..., "f1": ..., "acc": ..., "recall": ..., "count": ...},
      "S-Sparse": {...},
      "S-Dense": {...},
      "S-Hybrid": {...},
      "M": {...}
    },
    "action_distribution": {"Z": N, "S-Sparse": N, ...}
  },
  "overall": {"em": 0.xx, "f1": 0.xx, "acc": 0.xx, "recall": 0.xx, "count": N}
}
```

### 三、效率指标

#### 1. 推理速度
- 分类阶段记录每秒处理问题数 (`q/s`)，以及总推理耗时
- 体现路由机制将简单问题直接旁路（Z 动作）带来的效率提升

#### 2. 检索效率分析
- **平均检索次数**：各路由动作下的实际检索次数（Z=0, S=1, M=1~6）
- **多跳推理实际跳数 vs 最大跳数 (6)**：衡量提前终止的效率
- **查询重写成功率**：语义去重后有效查询的比例

#### 3. 检索召回率上界 (Upper Bound Analysis)
通过 `evaluate/upper_bound_analysis/` 进行上界分析：

**检索召回率 (Retrieval Recall)**：
- `Recall@5`、`Recall@20`、`Recall@100`：标准答案在 Top-K 检索结果中的命中率
- 退化分析：各 K 阈值间的答案丢失量

**阅读器上界 (Reader Upper Bound)**：
- 在提供完美检索结果的条件下，评估阅读器 (LLM) 的 EM/F1 上界
- 错误类型分类：空答案、提取错误、格式错误、推理错误
- 诊断建议基于 EM 阈值：
  - EM ≥ 0.70 → 瓶颈在检索
  - EM 0.50-0.70 → 先优化 Prompt，再优化检索
  - EM 0.30-0.50 → 模型能力偏弱，优化 Prompt/CoT
  - EM < 0.30 → 需更换更强模型
