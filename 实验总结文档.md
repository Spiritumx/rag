# RAG 路由分类器实验总结文档

## 项目概述

本项目实现了一个基于智能分类器的 RAG (Retrieval-Augmented Generation) 路由系统，通过预测问题的复杂度来自动选择最优的检索策略，提升问答系统的效率和准确性。

## 实验架构

```
问题输入 → 分类器 → 路由决策 → RAG策略执行 → 答案生成 → 评估
```

---

## 一、分类器训练模块 (classifier/)

### 1.1 数据生成 (classifier/data/)

#### 核心文件：
- **generate_training_data.py** - RAG 路由分类训练数据生成器
- **generate_L0_data.py** - L0 类别合成数据生成器
- **analyze_label_distribution.py** - 标签分布分析工具

#### 分类体系：

**三个维度的层级分类：**

1. **复杂度标签 (Complexity)**
   - **L0**: 无需检索（逻辑推理、代码生成、翻译、闲聊）
   - **L1**: 单跳检索（单一实体的事实查询）
   - **L2**: 多跳检索（比较、多实体关系、嵌套逻辑）

2. **索引策略 (Index Strategy)**
   - **None**: L0 专用
   - **Lexical**: 词法检索（BM25），适用于精确实体、专有名词
   - **Semantic**: 语义检索（向量搜索），适用于描述性、抽象查询
   - **Hybrid**: 混合检索，用于复杂 L2 问题

3. **执行动作 (Action)**
   - **Z**: 零检索（直接生成）
   - **S-Sparse**: 单跳稀疏检索（BM25）
   - **S-Dense**: 单跳密集检索（向量）
   - **S-Hybrid**: 单跳混合检索
   - **M**: 多跳推理（Multi-hop Agent）

#### 数据生成流程：

1. **基础训练数据生成**
   - 使用 OpenAI Structured Outputs API (gpt-4o-mini)
   - 数据集类型作为弱标签提示（单跳/多跳）
   - 系统 Prompt 包含详细的分类规则和示例
   - 并发处理 + 异步保存（最大并发：10）
   - 输出格式：
     ```json
     {
       "dataset": "hotpotqa",
       "question_id": "xxx",
       "question_text": "Who is older, Brad Pitt or Tom Cruise?",
       "reasoning": "比较两个实体的年龄，需要检索两者的出生日期...",
       "complexity_label": "L2",
       "index_strategy": "Hybrid",
       "action": "M"
     }
     ```

2. **L0 数据合成**
   - 六大类别：Coding, Logic Puzzles, Creative Writing, Chitchat, Text Processing, Common Sense
   - 批量生成（每批 10 条）
   - 与现有数据合并并随机打乱
   - 目标：600 条 L0 样本

3. **数据集分布**
   - 多跳数据集：2wikimultihopqa, hotpotqa, musique
   - 单跳数据集：nq, squad, trivia

### 1.2 模型训练 (classifier/train/)

#### 核心文件：
- **finetune.py** - 主训练脚本
- **evaluate.py** - 细粒度评估（5 个 Action 类别）
- **evaluate_coarse.py** - 粗粒度评估（Z/S/M 三类别）

#### 训练配置：

**基座模型**: Qwen2.5-3B-Instruct

**训练方法**: LoRA (Low-Rank Adaptation)
- Rank (r): 16
- Alpha: 16
- Dropout: 0
- 目标模块: q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj

**训练参数**:
- 优化器: adamw_8bit
- 学习率: 1e-4
- Batch size: 4 (per device)
- Gradient accumulation: 8
- Epochs: 2
- Scheduler: cosine
- 最大序列长度: 2048
- 混合精度: FP16/BF16

**训练框架**: Unsloth + TRL (SFTTrainer)

#### 系统 Prompt：
```
You are an expert RAG router. Analyze the user query complexity
and determine the optimal retrieval strategy.
Output the analysis in the following format:
Analysis: <reasoning process>
Complexity: <L0/L1/L2>
Index: <None/Lexical/Semantic/Hybrid>
Action: <Z/S-Sparse/S-Dense/S-Hybrid/M>
```

#### 评估指标：
- **粗粒度准确率**: Z vs S vs M 的三分类准确率（论文主结果）
- **细粒度指标**: 5 个 Action 类别的精确率、召回率、F1
- **混淆矩阵**: 识别误分类模式

## 二、端到端评估模块 (evaluate/)

### 2.1 三阶段评估流程

#### Stage 1: 分类阶段 (stage1_classify.py)

**功能**: 使用训练好的分类器对测试集问题进行批量分类

**关键特性**:
- 批量推理（默认 batch_size=16）
- 异步保存 checkpoint（每 5 个批次）
- 断点续传支持
- 速度监控（问题/秒）
- 错误回退机制（批次失败时逐条处理）

**输出**:
```json
{
  "question_id": "xxx",
  "question_text": "...",
  "predicted_action": "M",
  "full_response": "Analysis: ... Complexity: L2 ...",
  "dataset": "hotpotqa"
}
```

**性能优化**:
- GPU 批量推理
- 多线程保存
- 进度条显示（tqdm）

#### Stage 2: 生成阶段 (stage2_generate.py)

**功能**: 根据分类结果，将问题路由到相应的 RAG 策略并生成答案

**路由策略映射**:

| Action | 检索方法 | 检索后端 | 跳数 | 配置文件 |
|--------|---------|---------|-----|---------|
| Z | 无检索 | - | 0 | llama_baseline_no_retrieval.jsonnet |
| S-Sparse | BM25 | sparse | 1 | llama_onehop_sparse.jsonnet |
| S-Dense | Vector | dense | 1 | llama_onehop_dense.jsonnet |
| S-Hybrid | BM25+Vector | hybrid | 1 | llama_onehop_hybrid.jsonnet |
| M | 多跳 Agent | hybrid | 2-6 | M_core.py (Agentic) |

**执行流程**:
1. 加载 Stage 1 的分类结果
2. 按 Action 分组问题
3. 为每个组创建临时输入文件
4. 调用对应的推理配置
   - **M 策略**: 直接调用 Python 函数 `execute_real_multihop()`
   - **其他策略**: 使用 subprocess 调用 `configurable_inference` 模块
5. 合并所有预测结果
6. 保存答案、推理链、检索上下文

**并行化**:
- M 策略：ThreadPoolExecutor（可配置线程数）
- 其他策略：subprocess 支持多线程
- 动态超时计算：基于问题数和策略类型

**输出**:
- `predictions.json`: {question_id: answer}
- `predictions_chains.txt`: 推理过程
- `predictions_contexts.json`: 检索的文档

#### Stage 3: 评估阶段 (stage3_evaluate.py)

**功能**: 计算答案质量指标

**评估指标**:
- **EM (Exact Match)**: 精确匹配率
- **F1**: Token 级别的 F1 分数
- **ACC**: 准确率
- **Recall**: 召回率

**统计维度**:
- 整体性能
- 按数据集统计
- 按 Action 分组统计
- Action 分布统计

**输出文件**:
- `evaluation_metrics.json`: JSON 格式的结构化指标
- `detailed_report.txt`: 可读性强的文本报告

### 2.2 核心组件详解

#### M_core.py - 专家级多跳 Agent

这是整个系统最复杂的模块，实现了逻辑引导的多跳推理。

**架构**: Expert Agentic RAG (Logic-Guided)

**三阶段流程**:

**Stage 0: 逻辑规划 & 初始检索**
```
输入问题 → 逻辑规划器 → 生成执行计划 → 初始检索
```
- 使用 LLM 将自然语言问题转化为逻辑表达式
- 示例: "What is the capital of the country where A was born?"
  → Plan: `Birthplace(A) -> [Country], Capital([Country]) -> [Answer]`
- Warm Start: 使用原始问题进行初始检索（混合模式）

**Stage 1: 迭代推理循环**（最多 6 跳）
```
循环直到找到答案或达到最大跳数:
  1. LLM 生成 Thought 和 Action
  2. 验证 Action 合法性
  3. 执行检索 / 回答
  4. 更新上下文
```

**关键机制**:
- **物理截断**: 防止 LLM 自行生成多跳幻觉
- **强力清洗**: 移除对话式废话（"I need to find", "looking for" 等）
- **语义去重**: Jaccard 相似度检测循环查询（阈值 0.9）
- **证据校验**: 宽松模式，检测 "not found" 等否定信号
- **重试机制**: 每步最多 3 次重试，带反馈纠错

**Stage 2: 最终答案生成**
```
检索文档（最多40个） → 精简日志 → CoT Prompt → 生成最终答案
```
- 注入逻辑规划作为 Blueprint
- 注入完整的调查日志（Investigation Log）
- 提供所有检索文档作为证据

**Prompt 设计亮点**:
1. **逻辑策略注入**: 每次推理都看到逻辑规划
2. **历史去重**: 明确告知 "Past Actions" 避免重复
3. **格式强制**: 只输出关键词，不要对话式长句
4. **错误反馈**: 实时修正 LLM 的错误行为

**错误处理**:
- 超时保护（60s/90s）
- 异常捕获与日志记录
- 兜底回答: "I don't know"

#### 辅助工具模块 (evaluate/utils/)

1. **config_loader.py**: YAML 配置加载
2. **classifier_loader.py**:
   - 加载 LoRA 权重
   - 批量推理优化
   - 解析模型输出
3. **data_loader.py**: JSONL 数据集加载
4. **result_manager.py**:
   - 结果保存/加载
   - 断点续传
   - 文件路径管理
5. **service_checker.py**: 检查 Retriever 和 LLM 服务健康状态
6. **llama_generator.py**: 自动启动/停止 LLM 服务器

#### 配置映射 (configs/action_to_config_mapping.py)

动态选择 JSONNET 配置文件：
- 根据 Action 和 Dataset 选择对应配置
- 支持环境变量注入（RETRIEVER_HOST, LLM_SERVER_HOST 等）
- 配置存放路径: `evaluate/configs/llama_configs/`

### 2.3 主流程编排 (run_pipeline.py)

**PipelineRunner 类**:
- 协调三个阶段的执行
- 可选择运行部分阶段（`--stages 1 2 3`）
- 可选择部分数据集（`--datasets hotpotqa musique`）
- 自动服务健康检查
- 自动启动 LLM 服务器（可配置）
- 统一日志管理

**命令行示例**:
```bash
# 运行完整流程
python evaluate/run_pipeline.py

# 只运行分类
python evaluate/run_pipeline.py --stages 1

# 运行生成和评估（跳过分类）
python evaluate/run_pipeline.py --stages 2 3 --datasets hotpotqa

# 使用自定义配置
python evaluate/run_pipeline.py --config my_config.yaml

# 跳过服务检查
python evaluate/run_pipeline.py --no-service-check
```

## 三、主要技术方法

### 3.1 数据生成方法

**OpenAI Structured Outputs**:
- 使用 Pydantic Schema 强制输出格式
- 确保字段类型安全（Literal 枚举）
- 自动重试 + 指数退避

**弱标签提示 (Weak Label Hinting)**:
- 数据集类型（单跳/多跳）作为 Context Hint
- 提示 LLM 参考但不强制遵守
- 减少标注偏差

**数据增强策略**:
- 合成 L0 数据平衡类别分布
- 随机打乱防止过拟合
- 多样性保证（6 个 L0 子类别）

### 3.2 模型优化方法

**LoRA 微调**:
- 参数高效（只训练 0.1% 的参数）
- 快速迭代
- 易于部署（可热插拔）

**Unsloth 加速**:
- 2x 训练速度提升
- 内存优化（Gradient Checkpointing）
- Flash Attention 支持

**混合精度训练**:
- 自动检测硬件能力（BF16 优先）
- 降低显存占用
- 保持数值稳定性

### 3.3 推理优化方法

**批量推理**:
- GPU 批处理（16 个问题/批次）
- 显著降低延迟（从 10s → 0.5s per question）

**断点续传**:
- 结果文件实时保存
- 意外中断后无需重新开始
- 适用于大规模评估

**并行化**:
- M 策略的多线程并行
- 其他策略的 subprocess 并行
- 充分利用多核 CPU

### 3.4 多跳推理方法

**逻辑规划 (Logical Planning)**:
- 将问题分解为结构化步骤
- 指导后续搜索方向
- 减少盲目探索

**迭代检索 (Iterative Retrieval)**:
- 动态生成下一跳查询
- 逐步积累上下文
- 避免信息爆炸

**证据校验 (Evidence Verification)**:
- 检查是否有支持文档
- 检测 LLM 的不确定性
- 防止无根据的回答

**循环检测 (Loop Detection)**:
- 语义相似度去重
- 阻止重复查询
- 强制探索新方向

## 五、后端服务实现

### 5.1 检索服务器 (retriever_server/)

检索服务器是整个 RAG 系统的核心组件，提供多种检索策略的统一接口。

#### 架构组件

**serve.py** - FastAPI 服务入口
- 端点：`POST /retrieve/`
- 自动加载和管理多个检索模型
- 环境变量配置支持

**模型配置**：
- **Dense Model** (密集向量)
  - 默认：`all-MiniLM-L6-v2` (sentence-transformers)
  - 用途：HNSW 向量检索
  - 维度：384

- **SPLADE Model** (稀疏向量)
  - 默认：`splade-v3` (naver)
  - 用途：学习型稀疏检索
  - 输出：稀疏 token-weight 字典

- **Reranker Model** (重排序)
  - 默认：`cross-encoder-ms-marco-MiniLM-L-6-v2`
  - 用途：二阶段精排
  - Top-K：50（可配置）

**unified_retriever.py** - 统一检索接口
- 封装 `ElasticsearchRetriever`
- 提供统一的 API：
  ```python
  retrieve_from_elasticsearch(
      query_text: str,
      max_hits_count: int = 3,
      retrieval_backend: str = "bm25",  # bm25|hnsw|splade|hybrid
      corpus_name: str = "hotpotqa",
      rerank_query_text: str = None     # 独立的 rerank 查询
  )
  ```

**elasticsearch_retriever.py** - 核心检索实现

支持 **四种检索后端**：

1. **BM25 (词法检索)**
   ```
   Elasticsearch Match Query
   → 基于 TF-IDF 的经典检索
   → 适用于精确实体匹配
   ```

2. **HNSW (密集向量检索)**
   ```
   Query Embedding → HNSW Index → Cosine Similarity
   → 基于语义相似度
   → 适用于描述性查询
   ```

3. **SPLADE (学习型稀疏检索)**
   ```
   MLM Model → Sparse Vector → Rank Features Query
   → 结合词法和语义
   → 比 BM25 更智能
   ```

4. **Hybrid (混合检索)** ⭐
   ```
   Phase 1-3: 并行检索
     - BM25 召回
     - HNSW 召回
     - SPLADE 召回

   Phase 4: 加权融合 + Reranker
     - 分数归一化
     - 权重融合 (可配置)
     - CrossEncoder 重排序
   ```

**黄金三角策略** (Golden Triangle):
- **检索阶段**：使用短查询（生成的关键词）
  - 提高召回率
  - 减少噪音
- **重排序阶段**：使用长查询（原始问题 + 桥梁实体）
  - 利用更多上下文
  - 提升精排准确性

**build_index.py** - 索引构建工具

功能：
- 批量索引多数据集（hotpotqa, musique, 2wikimultihopqa, iirc, wiki）
- 支持混合索引（BM25 + HNSW + SPLADE）
- 断点续传（wiki 数据集，210万文档）
- 批量 embedding 生成

索引结构：
```json
{
  "title": "text (BM25)",
  "paragraph_text": "text (BM25)",
  "dense_embedding": "dense_vector (HNSW, 384维)",
  "splade_vector": "rank_features (稀疏向量)",
  "is_abstract": "boolean",
  "paragraph_index": "integer"
}
```

命令示例：
```bash
# 构建混合索引
python retriever_server/build_index.py hotpotqa \
  --use-dense \
  --use-splade \
  --batch-size 32 \
  --es-chunk-size 2000

# Wiki 数据集（支持断点续传）
python retriever_server/build_index.py wiki \
  --use-dense \
  --enable-checkpoint \
  --checkpoint-interval 1000
```

#### 服务启动

```bash
# 设置环境变量
export RETRIEVER_DENSE_MODEL_PATH="./models/dense/all-MiniLM-L6-v2"
export RETRIEVER_SPLADE_MODEL_PATH="./models/splade/splade-v3"
export RETRIEVER_RERANKER_MODEL_PATH="./models/reranker/cross-encoder-ms-marco-MiniLM-L-6-v2"
export RETRIEVER_ES_HOST="http://localhost"
export RETRIEVER_ES_PORT=9200
export RETRIEVER_PORT=8001

# 启动服务
python retriever_server/serve.py
```

### 5.2 LLM 服务器 (llm_server/)

LLM 服务器提供文本生成能力，支持单个和批量推理。

#### 核心文件

**serve_llama.py** - 基础 Llama 服务器

特性：
- **单推理端点**: `GET/POST /generate`
- **批量推理端点**: `POST /generate_batch`
- **Left Padding**: 批量生成时的关键优化
- **FP16 混合精度**: 降低显存占用

关键配置：
```python
model = AutoModelForCausalLM.from_pretrained(
    MODEL_PATH,
    device_map="auto",           # 自动设备分配
    torch_dtype=torch.float16,   # 半精度
    trust_remote_code=True,
    local_files_only=True        # 离线模式
)

tokenizer.padding_side = 'left'  # 批量推理关键
```

API 示例：
```bash
# 单个推理
curl "http://localhost:8000/generate?prompt=Hello&max_length=200"

# 批量推理
curl -X POST http://localhost:8000/generate_batch \
  -H "Content-Type: application/json" \
  -d '{
    "prompts": ["Hello", "How are you", "Tell me a joke"],
    "max_length": 200
  }'
```

**serve_llama_autobatch.py** - 自动批处理服务器 ⭐

高级特性：
- **动态批处理**: 自动聚合请求
  - `MAX_BATCH_SIZE=8`: 最大批次大小
  - `BATCH_TIMEOUT_MS=50`: 等待超时（毫秒）
- **参数分组**: 相同参数的请求批量处理
- **后台 Worker**: 独立线程处理批次
- **Llama-3 Chat Template**: 标准化输入格式

```python
def apply_llama3_template(raw_content: str) -> str:
    """Llama-3 对话模板"""
    return (
        "<|begin_of_text|>"
        "<|start_header_id|>system<|end_header_id|>\n\n"
        "You are a helpful AI assistant."
        "<|eot_id|>"
        "<|start_header_id|>user<|end_header_id|>\n\n"
        f"{raw_content}<|eot_id|>"
        "<|start_header_id|>assistant<|end_header_id|>\n\n"
    )
```

批处理流程：
```
请求队列 → Worker 收集批次 → 参数分组 → 批量推理 → 返回结果
   ↓           ↓                  ↓             ↓
 Queue    (最多8个,           同参数合并    GPU 加速
          50ms超时)
```

优势：
- **吞吐量提升**: ~4-8x（取决于批次大小）
- **延迟平衡**: 50ms 超时保证响应速度
- **资源利用**: 充分利用 GPU 并行能力
